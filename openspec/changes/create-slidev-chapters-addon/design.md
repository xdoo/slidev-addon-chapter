## Context

Slidev's built-in `<Toc />` is title-oriented: it derives entries from slide titles and title levels. The requested model instead introduces chapter boundaries that span several slides and must be reusable independently of any theme. Slidev 52.18.0 documentation describes addons as the place for separately usable functionality and components, while themes focus on appearance; it also discourages addons from global style and layout overrides.

The official syntax transports YAML objects in per-slide frontmatter and supports imported slide files. The directory conventions auto-discover `components/*` and make all customization directories optional. The official addon guide confirms local preview through `addons: [./]`, direct publication of `.vue`/`.ts`, the `slidev-addon-` naming convention, and Slidev engine constraints. Public runtime documentation exposes current-slide state through `@slidev/client` (including `useNav()` / `currentPage`) and explicitly warns against nested private imports. It does not, in the material reviewed for this proposal, establish a stable documented API for enumerating every resolved slide and its custom frontmatter; that is the principal implementation spike.

Normative references are the current official pages for [Writing Addons](https://sli.dev/guide/write-addon), [Theme and Addons](https://sli.dev/guide/theme-addon), [Writing Themes](https://sli.dev/guide/write-theme), [Directory Structure](https://sli.dev/custom/directory-structure), [Syntax](https://sli.dev/guide/syntax), [Built-in Components](https://sli.dev/builtin/components), [Built-in Layouts](https://sli.dev/builtin/layouts), [Global Context](https://sli.dev/guide/global-context), [Building and Hosting](https://sli.dev/guide/hosting), and the [Slidev repository](https://github.com/slidevjs/slidev). Community structure was compared, non-normatively, with [window-mockup](https://github.com/whitphx/slidev-addon-window-mockup), [qrcode](https://github.com/kravetsone/slidev-addon-qrcode), [bpmn](https://github.com/emaarco/slidev-addon-bpmn), [d2-diagrams](https://github.com/ndisidore/slidev-addon-d2-diagrams), and [tldraw](https://github.com/AlbertBrand/slidev-addon-tldraw).

## Goals / Non-Goals

**Goals:**

- Define chapter semantics, validation, ordering, slide ranges, and behavior before the first declaration.
- Expose a small typed `useChapters()`-style API and a theme-neutral `<ChapterToc />`.
- Work in presentation, presenter, direct-load/refresh, static build, and documented import scenarios.
- Keep extraction as pure TypeScript logic and isolate Slidev integration behind a narrow adapter.
- Use only documented public APIs and add infrastructure only after the spike demonstrates a need.
- Establish concrete unit, integration, theme, build, PDF/PPTX, and documentation checks.

**Non-Goals:**

- Corporate design, global typography, logos, footer, or chapter-divider appearance.
- Providing or overriding layouts, overriding theme styles/configuration, or general Slidev navigation.
- `<ChapterIndicator />`, `<ChapterProgress />`, previous/next chapter controls, or a nested slide TOC in the first release.
- Implementing production code in this planning change.
- Promising export interactivity: PDF/PPTX output must remain readable, but links are verified only where the exporter preserves them.

## Decisions

### 1. `chapter` defines semantics; `layout` only defines presentation

A slide whose frontmatter contains a valid `chapter` object starts a chapter. It belongs to that chapter, as do subsequent slides through the slide before the next declaration. `layout: chapter` is ignored by extraction. It can coexist with `chapter`, but neither implies the other.

This supports both requested declaration forms: ordinary content slides can begin chapters, and themes can supply a visual separator layout. Coupling semantics to `layout: chapter` was rejected because layouts are theme-owned, the layout may not exist in every theme, and a theme may use that name without intending addon semantics. Inferring chapters from headings or `<Toc />` was rejected because it recreates slide-level structure rather than explicit chapter intent.

### 2. Use a pure extractor plus a thin Slidev adapter

The core accepts normalized inputs conceptually shaped as `{ slideNumber, frontmatter }[]` and returns immutable ordered chapter records plus diagnostics. A runtime adapter obtains resolved slides and current page through documented Slidev APIs. This preserves deterministic unit testing and contains version coupling.

The API spike must compare documented alternatives before selecting one:

- Runtime enumeration from a public `@slidev/client` export: simplest and reactive if it exposes all resolved slides and custom frontmatter.
- A documented virtual module generated by Slidev: suitable if explicitly public and stable for addons.
- A documented setup hook or transformer that injects normalized metadata: acceptable only if runtime enumeration is unavailable.

Private nested imports, undocumented generated globals, and direct parsing of Slidev internals are rejected. A custom Vite plugin, Markdown transformer, pre-parser, or `setup/` entry is not part of the baseline; the spike may propose one only with an official public contract and a written trade-off.

### 3. Require explicit IDs in the first release

`chapter.title` and `chapter.id` are required non-empty strings; IDs must be unique in the resolved presentation. Explicit IDs give stable links, predictable imports, and actionable validation. Automatically slugifying titles is convenient but creates locale/algorithm questions, collisions, and link changes when copy changes. Deterministic title-derived IDs remain a possible later opt-in after collision and compatibility policy is specified.

A repeated ID is an error even if title and object contents are identical; declarations are boundaries, not references. This prevents accidental duplicate chapters from being silently collapsed. Invalid definitions produce understandable diagnostics with slide number and field; development should fail loudly. Exact build-time versus render-time failure behavior depends on the integration API and is resolved by the spike, but invalid data must never yield ambiguous chapter state.

### 4. Slides before the first chapter remain unassigned

Pre-chapter slides are allowed and are not represented as a synthetic chapter. Consequently `currentChapter` is `undefined` on those slides. This preserves the rule that each displayed chapter comes from an explicit declaration and avoids inventing a title/ID. A future configurable preface chapter can be added without changing explicit chapter semantics.

### 5. Public API is small, readonly, and slide-number based

The target contract is:

```ts
interface Chapter {
  readonly id: string
  readonly title: string
  readonly index: number
  readonly startSlide: number
  readonly endSlide: number
  readonly slideNumbers: readonly number[]
}

interface ChapterState {
  readonly chapters: readonly Chapter[]
  readonly currentChapter?: Chapter
}
```

`useChapters()` returns reactive readonly state or the closest idiomatic equivalent supported by the selected public Slidev/Vue APIs. Indices are zero-based library indices; display numbering is one-based. Slide numbers follow Slidev's public one-based convention. Final reactive wrappers, route/link types, and error surface are deliberately gated by the spike.

The adapter derives `currentChapter` from the current public slide number and inclusive `[startSlide, endSlide]`, so direct URLs and refreshes do not depend on navigation history. The same computation must be used in regular and presenter contexts.

### 6. `<ChapterToc />` is an accessible, minimally styled consumer

Initial props are `showNumbers?: boolean` and `highlightCurrent?: boolean`, both defaulting to `false` unless usability testing during implementation justifies a documented default change. Each chapter renders once, in deck order, with a link to `startSlide`. It consumes the public chapter API instead of re-extracting metadata. The current entry exposes semantic state such as `aria-current` and a stable class/data hook while avoiding global CSS; consumers and themes control appearance.

Navigation should use a documented Slidev navigation/router facility that generates base-path-safe links and works after static build. Hand-built URLs are a fallback only if officially documented. Export output renders the ordered labels/numbers without relying on click handlers.

### 7. Minimal conventional package structure

The implementation starts with `components/ChapterToc.vue`, a public composable/module, a separate pure extractor, tests, `slides.md`, `package.json`, and `README.md`. Exact module placement/export mapping is finalized after verifying whether Slidev auto-imports addon composables or only components. `layouts/`, `styles/`, `setup/`, `vite.config.ts`, and `uno.config.ts` are excluded unless demonstrated necessary.

`slidev-addon-chapters` matches the documented full package prefix and is the recommended working name. Documentation should show the full name for clarity even where Slidev accepts shorthand. `package.json` will carry `slidev` and `slidev-addon` keywords, semantic versioning, and a tested `engines.slidev` floor selected from the spike rather than guessed.

### 8. Layered verification

- Unit tests cover extraction, ranges, one-slide chapters, pre-chapter slides, duplicate IDs, missing ID/title, and current-chapter lookup.
- The local `slides.md` uses the officially documented `addons: [./]` and demonstrates ordinary chapter starts, optional theme layout use, preface slides, and multiple-length chapters.
- Integration checks start dev, confirm component auto-registration, verify chapter-only output/navigation/direct refresh/presenter behavior, run static `slidev build`, and exercise default plus one maintained second theme.
- PDF and PPTX export behavior is checked and documented separately: readable content is required; navigation/link preservation is recorded rather than assumed.
- `test`, `typecheck`, and `build` are required scripts/checks. A narrow Playwright smoke test is added in the first change only if needed to reliably cover route refresh, presenter, and navigation; broad cross-browser/export automation may follow.

### 9. Single Source of Truth

Chapters are defined exactly once. All chapter overviews, agendas, and navigation components are generated automatically from the chapter definitions. Manual duplication of chapter information is intentionally avoided.

### 10. Chapter IDs MUST be unique within a presentation.

Each chapter represents a unique structural element of the presentation. Its ID serves as the stable identifier for navigation, generated agendas, tables of contents, progress indicators, and future extensions. Duplicate chapter IDs are not allowed and must result in a validation error.

## Risks / Trade-offs

- [No clearly documented all-slide metadata runtime API is confirmed] → Make API discovery the first blocking spike; record source links and a minimal proof before implementation.
- [A public API may differ across supported Slidev versions] → Choose and document a tested minimum version, keep the adapter isolated, and test against the supported range where practical.
- [Imported slides may merge/normalize frontmatter differently] → Test imported files through the same public resolved-slide source and declare unsupported cases rather than reparsing source files.
- [Custom frontmatter typing may not support module augmentation] → Verify official types; otherwise validate an `unknown` boundary locally and export addon-owned types without patching Slidev internals.
- [Component auto-registration may not imply composable auto-import] → Auto-register only `ChapterToc`; expose `useChapters` via documented package exports if composable conventions are not guaranteed.
- [Presenter and print contexts can expose different routing/reactivity] → Derive state from public current-page data and run context-specific integration/export checks.
- [Strict explicit IDs add authoring friction] → Prefer stable contracts in v1 and document a potential opt-in slug policy as a follow-up.
- [Rendering `slideNumbers` duplicates range information] → Keep it for the requested ergonomic API in v1; implementation may compute/freeze it once and revisit only through a versioned API change.

## Migration Plan

This is a new addon with no data migration. Implementation proceeds behind a pre-1.0 package version: complete the API spike, establish the minimum Slidev version, implement and verify the contract, document usage and limitations, then publish under semantic versioning. Rollback consists of removing the addon from deck headmatter/package dependencies; existing `chapter` custom frontmatter remains inert to Slidev and themes.

## Open Questions

1. Which documented public Slidev API exposes every resolved slide's custom frontmatter at runtime, including `src` imports, and is it available identically in dev, build, presenter, and print contexts?
2. Does the public slide record retain arbitrary nested `chapter` data with a stable type, and is module augmentation officially supported for per-slide frontmatter?
3. Is `useNav().currentPage` the supported current-page source in every required context, and what public navigation primitive should generate base-path-safe links to a slide?
4. Are addon `components/` auto-registered in all required build modes exactly as project components are, and must `useChapters()` be explicitly imported from a package export?
5. Can the minimal addon remain transformer- and setup-free, or does the chosen public metadata source require a documented hook?
6. How do current PDF and PPTX exporters preserve anchors/links and presenter-only state, and what parts can be automated reliably?
7. Which currently supported Slidev version becomes the initial engine floor after the proof, and which maintained second theme is used for compatibility testing?
